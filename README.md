# pip-audit-scan-action

Composite GitHub Action that runs **[pip-audit](https://github.com/pypa/pip-audit)** against a **`requirements.txt`** (or similar `-r` file) and reports **vulnerability** findings (dependency issues) using the [PyPA advisory database](https://github.com/pypa/advisory-database) via PyPI’s metadata. Output uses **`vuln_count`**.

## Behavior

- **`vuln_count = 0` and pip-audit succeeds** → `scan_status=clean`, step succeeds.
- **`vuln_count > 0`** → `scan_status=vulnerabilities_found`. If **`fail_on_vuln`** is **`true`**, the step exits **1**; if **`false`**, the step exits **0** after logging the count (useful for reporting-only jobs).
- **pip-audit errors or empty JSON** → `scan_status=scanner_error`, step exits with pip-audit’s exit code.

The action always runs pip-audit once with **`--format json`** internally to compute **`vuln_count`**. If **`format`** is not **`json`**, it runs pip-audit again to produce the human-readable or SBOM output you asked for (or to write **`report_file`**). That **second** pass is **cost/latency only** (no correctness issue); a future optimization could reduce duplicate work.

## Caller responsibilities

- Use a **Linux** runner (same as other composite actions using `bash`); **`actions/setup-python`** is invoked inside the action. Inputs are **validated before** pip-audit is installed so missing requirements files fail fast.
- **`requirements_file`** must exist under **`working_directory`** before the action runs (the action validates this).
- Pin **`pip_audit_version`** if you want reproducible CI as pip-audit and the advisory data evolve.

### Pip hash pinning

For the **default** **`pip_audit_version`** (**`2.7.3`**), the action installs from **`constraints/pip-audit-2.7.3.txt`** with **`pip --require-hashes`**. Other versions log a **`::warning`** and use an unhashed **`pip install`**. After bumping the default in **`action.yml`**, run **`scripts/refresh-pip-constraints.sh`** and commit the updated **`constraints/*.txt`**.

## Inputs

| Input | Default | Description |
|--------|---------|-------------|
| `python_version` | `3.11` | Python version for `actions/setup-python`. |
| `pip_audit_version` | `2.7.3` | Exact pip-audit version; default uses **`pip --require-hashes`** when **`constraints/pip-audit-2.7.3.txt`** is present. |
| `working_directory` | `.` | Directory that contains the requirements file. |
| `requirements_file` | `requirements.txt` | Path relative to **`working_directory`** (passed to `pip-audit -r`). |
| `format` | `columns` | Report format: `columns`, `json`, `cyclonedx-json`, `cyclonedx-xml`, or `markdown`. |
| `report_file` | *(empty)* | If set, write the rendered report to this path relative to **`working_directory`** instead of only printing to the log. |
| `fail_on_vuln` | `true` | If `true`, exit **1** when any vulnerability is found (default fail-closed). Set **`false`** for reporting-only jobs. |

## Outputs

| Output | Description |
|--------|-------------|
| `vuln_count` | Number of vulnerability records across dependencies in the JSON report. |
| `scan_status` | `clean`, `vulnerabilities_found`, or `scanner_error`. |

## Example

```yaml
jobs:
  deps:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: thadiust/pip-audit-scan-action@main
        with:
          working_directory: "."
          requirements_file: "requirements.txt"
          python_version: "3.11"
          format: "columns"
          fail_on_vuln: true
```

With **`report_file`** (for example for artifacts):

```yaml
      - uses: thadiust/pip-audit-scan-action@main
        with:
          requirements_file: "requirements.txt"
          format: "cyclonedx-json"
          report_file: "sbom.cdx.json"
          fail_on_vuln: true
```

## Scope and limits

- Scans **declared** dependencies in the requirements file, not necessarily every package in a full environment lockfile unless you point **`requirements_file`** at a **fully pinned** file (for example generated by **`pip-compile`**). Unpinned ranges can resolve differently over time.
- Results depend on the **accuracy and coverage** of the advisory database and PyPI metadata; they may **not** include every known vulnerability affecting your stack.
- **Remediation** is up to you: upgrade, replace, or justify risk; the action does not auto-fix.

For a full Python security pipeline (Gitleaks, Bandit, pip-audit), see [`thadiust/workflow-python`](https://github.com/thadiust/workflow-python).
